# My Skills

这是一个用于持续积累个人 Codex skills 的仓库。

它不只服务于当前这两个写作 skill，而是作为一个长期演进的 skill 集合，逐步沉淀可复用的写作、研究、知识整理、工程协作和工作流能力。

当前已经落地的是中文长文写作方向的第一批 skill，后续会继续扩展到更多类型。

## 仓库目标

- 让常用 skill 以版本化方式持续迭代，而不是散落在临时对话里
- 把可复用的提示约束、流程、参考文档沉淀成可维护资产
- 让不同 skill 有清晰边界，避免一个 skill 过宽、过糊、过度抢触发
- 为后续新增 skill 保留统一结构，方便扩展、迁移和共享

## 当前已收录的 skill

目前仓库里已经包含两类中文长文写作 skill：

### `write-opinion-article`

位置：`.agents/skills/write-opinion-article`

用于带明确判断的评论、专栏、观察文章。它强调作者立场、段落推进和评论感，更适合“我怎么看”这一类写作任务。

典型场景：

- 写评论稿
- 写专栏稿
- 基于文档、现象或事件写观察文章
- 需要保留作者判断和观点锋芒的长文

### `write-knowledge-article`

位置：`.agents/skills/write-knowledge-article`

用于朴实、专业、信息密度高的知识分享文章。它更强调讲清楚、讲具体、保留关键操作细节，更适合“把事情讲清楚”这一类写作任务。

典型场景：

- 写公众号知识分享
- 写教程或经验总结
- 基于官方文档写专业解读
- 需要少空话、少姿态、更多方法和边界条件的文章

## 当前写作 skill 的分工

- `write-opinion-article`：偏评论、偏判断、偏作者声音
- `write-knowledge-article`：偏讲解、偏专业、偏信息密度

当前这两个 skill 是互补关系，而不是互相替代：

- 如果重点是“我怎么看”，优先用 `write-opinion-article`
- 如果重点是“把事情讲清楚”，优先用 `write-knowledge-article`
- 如果后续出现新的写作类型，也应继续拆分，而不是不断往现有 skill 里堆逻辑

## 未来扩展方向

这个仓库后续会继续收录更多 skill，方向不限于写作，可能包括：

- 知识整理与内容生产
- 技术研究与资料解读
- 工程协作与代码工作流
- 自动化任务与日常运营支持

原则是：每个 skill 只解决一类相对明确的问题，保持边界清楚、触发自然、说明可维护。

## 目录结构

```text
.agents/skills/
├── <skill-name>/
│   ├── SKILL.md
│   ├── agents/openai.yaml
│   └── references/
└── ...
```

当前仓库中的 skill 也遵循这套结构，只是目前数量还不多。

## Skill 组织方式

每个 skill 默认包含：

- `SKILL.md`：skill 的触发描述和核心工作流
- `agents/openai.yaml`：UI 文案和默认 prompt
- `references/`：只在必要时加载的辅助约束、清单和说明

如果后续需要更强的确定性，也可以按需增加：

- `scripts/`：可执行脚本
- `assets/`：模板、素材或其他输出资源

## 使用方式

把本仓库中的 skill 目录放到 Codex 可发现的位置即可使用。当前仓库按项目内 skill 结构组织，主要面向：

- 仓库内共享 skill：`.agents/skills/`
- 随项目一起版本管理和迭代

在实际对话里，可以显式提到 skill 名称，例如：

- `使用 $write-opinion-article 写一篇评论稿`
- `使用 $write-knowledge-article 写一篇知识分享文章`

## 维护原则

- skill 尽量小而清楚，不追求大而全
- 能拆就拆，避免一个 skill 同时覆盖多种不相容任务
- 参考文档只保留真正能提高结果质量的约束和检查项
- 优先收敛触发边界，减少误用和误触发
- 当前 skill 是起点，不是终点；仓库结构应始终服务于后续扩展
