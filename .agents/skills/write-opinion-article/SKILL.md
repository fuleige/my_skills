# Write Opinion Article

## Overview

先把任务理解成“完成一篇能发出去的文章”，不要把它理解成“整理信息”或“生成说明”。默认交付完整正文；只有用户明确要求时，才输出提纲、要点或分节说明。

把文章写成有人在说话，而不是系统在归纳。保持明确立场、自然节奏和可读的段落推进，避免提纲腔、总结腔、汇报腔和“AI 平均语气”。

默认输出中文正文。除非用户明确要求其他语言，否则标题、导语、正文、结尾和备选版本都必须写成自然中文，而不是英式句法的中文翻译腔。

默认把文章写成“有判断，也有材料”的东西。不要让文章只剩观点姿态和知识点说明；要让读者看见具体场景、真实代价、细节抓手和论述落点。

## Workflow

### 1. 先确定文章合同

先识别以下关键信息：

- 主题是什么
- 想说服谁，或想让谁读完后产生什么感觉
- 文章类型是什么：评论、专栏、观察、人物式随笔、行业判断、热点观点、品牌署名文章
- 长度大概是多少，是否存在明确的发布约束
- 用户是否已经给出立场、素材、事实边界或禁区

如果信息不完整，但仍可继续，就做最小假设并直接推进。只在事实边界不清、容易误写结论时才回问用户。

### 2. 先找到“要表达的判断”，再开始写

先用一句话写出文章核心判断。这个判断必须是可争论、可辩护、可展开的，不要写成空泛正确话。

判断形成后，再补下面三项：

- 这篇文章真正回应的矛盾、误解或时代情绪是什么
- 读者为什么现在要读它，而不是以后再读
- 作者的观察站位是什么，凭什么有资格这样判断

不要写成“有利有弊”“值得关注”“未来可期”这类没有方向的平衡句。没有立场的文章，最后只会像综述。

### 3. 先补“材料层”，不要只靠观点硬撑

在搭段落之前，先确认这篇文章能落到哪些具体材料上。优先从已知事实、用户素材、常识范围内的真实情境里，补足至少 2 到 4 类抓手：

- 一个具体场景、动作或细节
- 一种常见误解，或一句值得反驳的话
- 一个真实代价：谁在承担、怎么承担
- 一个时间变化：以前怎样，现在怎样
- 一个对比：台面上怎么说，现实里怎么发生
- 一个角色视角：普通人、从业者、管理者、消费者、家长、创作者
- 一个能把抽象问题落地的小例子或典型现象

判断一旦离开这些材料，就容易滑成“知识分享”或“正确废话”。如果手里材料太少，不要用大而空的背景介绍把篇幅填满；要缩小论点，换成更窄但更真的写法。

默认让每个核心判断至少回答下面问题里的两个：

- 它具体发生在什么场景里
- 谁最先感受到它
- 它和过去相比变了什么
- 它带来的代价、尴尬或后果是什么
- 哪个细节最能说明它不是空话

### 4. 先搭段落推进，再搭提纲

优先按论述推进组织文章，不要默认用列点式小标题拆开。除非用户要求“分点写”“做成攻略”或题材本身就是清单文，否则尽量保持连续叙述。

不要默认为了不同平台切换不同腔调。除非用户明确要求适配某个平台，否则把精力放在内容本身：判断是否成立，语言是否自然，情绪是否有人味。

优先使用下面的推进方式之一：

1. 先抛出现象或冲突，再亮出判断
2. 先给出判断，再解释它为什么成立
3. 先写个人观察或具体场景，再抬升到更大的问题
4. 先反驳一种常见说法，再建立自己的版本

让每一段只完成一个动作，例如：

- 提出问题
- 扭转预期
- 补充证据
- 推进判断
- 拉高意义
- 收束余味

如果准备写提纲，也要写成段落动作，不要只写名词标签。

默认优先使用这些更容易写实、写厚的推进组合：

1. 具体现象 -> 作者判断 -> 为什么很多人误读了它
2. 场景或细节 -> 背后的结构性变化 -> 它带来的代价
3. 常见说法 -> 反驳 -> 更贴近现实的解释
4. 作者观察 -> 推开到更大问题 -> 回到普通人的感受

### 5. 起草时保持“人写的声音”

写首段时，优先从下面四种入口里选一种：

- 一个反常识判断
- 一个具体场景或细节
- 一个已经发生但多数人没说透的变化
- 一个读者熟悉但被误读的问题

正文写作时，持续做到下面几件事：

- 明确表达判断，不要一直躲在“有人认为”“某种程度上”后面
- 让句长有变化，避免每句都一样平
- 多用具体动词和具体情境，少用空抽象名词
- 让段与段之间存在推动关系，而不是并排堆放
- 尽量让抽象判断后面紧跟一个可感知句子，不要连续几句都停留在概念层
- 优先把“是什么”压缩到最低限度，把篇幅留给“为什么会这样”和“这意味着什么”
- 写到关键段落时，补上人物位置、现实摩擦、利益分配或情绪反应，而不是只补定义
- 保留适度主观性，但不要凭空造事实
- 允许有锋芒，但不要为了姿态而夸张失真

默认不要把文章写成下面这些样子：

- 开头先定义概念，再平铺背景
- 花大段篇幅做基础知识科普，真正的判断却很薄
- 每段都像“首先、其次、再次、最后”
- 结尾强行上价值
- 像行业汇报、产品方案、读书摘要或会议纪要
- 过度工整到失去说话感

如果用户要求第一人称，就大胆使用“我”；如果用户没有要求，也可以在确有作者观察时少量使用第一人称，但不要滥用自我表演。

### 6. 改写成可发布版本

初稿完成后，不要立刻交付。先进行一轮编辑式改写：

- 删掉所有不提供信息量的过渡句
- 把空话替换成具体判断
- 把“观点句”后面没有承接材料的地方补实
- 把重复表达合并
- 检查每一段是否真的推动了文章，而不是重复上一段
- 检查是否存在“讲了很多道理，但没有写出任何现实摩擦”的段落
- 检查文章是否把太多篇幅浪费在常识背景，而不是判断、细节和代价上
- 检查标题、开头、结尾是否属于同一篇文章
- 检查是否出现“像 AI 在总结”的句型

当草稿明显带有提纲腔、解释腔或模型腔时，先读 [references/human-voice.md](references/human-voice.md)。当文章有观点但发空、偏知识分享、缺细节支撑时，再读 [references/content-density.md](references/content-density.md)。交稿前再读 [references/publish-checklist.md](references/publish-checklist.md) 做最终检查。

### 7. 默认交付方式

默认直接给出完整文章。除非用户要求，不要在正文前加长篇说明，也不要在正文后附一串分析性 bullet points。

默认交付顺序如下：

1. 标题
2. 正文

标题默认写得清楚、准确、克制，不故意制造悬念，不追求标题党式刺激。只有在确实能提升成稿质量时，才额外补充副标题、导语或备选标题。不要把“写作过程”暴露给最终读者。

## Hard Rules

- 默认用中文写完整文章；不要混入英文段落、英文小标题或翻译腔表达，除非用户明确要求。
- 标题优先说明白文章在谈什么，不要靠夸张、反问、故作惊人来吸引点击。
- 不要把文章写成答题卡、百科词条或 PPT 备注。
- 不要用列点代替论述，除非用户明确要清单文。
- 不要机械追求“客观中立”；观点文章必须有自己的判断。
- 不要编造事实、引用、数据或经历。
- 不要把文章主体写成知识梳理、概念介绍或方法总结，除非用户明确要知识分享文。
- 不要让核心判断悬空；重要观点后面要尽量跟上场景、代价、细节、对比或角色视角。
- 不要在材料不够时拿宏大背景、套话分析和抽象名词硬凑篇幅；宁可缩小论点。
- 不要为了适配想象中的平台风格，把文章写成统一模板或流量文腔。
- 不要用套路化结尾，例如“归根结底”“总而言之”“在这个快速变化的时代”。
- 不要为了显得深刻而堆抽象大词。
- 不要把所有段落写成同样长度、同样句式、同样语气。
- 不要只追求“观点锋利”；还要保留人的温度、经验感和对读者的正常说话方式。

## Revision Triggers

当出现下面任一情况时，主动重写而不是只做轻修：

- 全文读起来像说明文，不像文章
- 全文读起来像知识分享，不像观点写作
- 观点存在，但没有段落推进
- 判断存在，但缺少细节、场景、代价或人物位置
- 每段都对，但合起来没有作者人格
- 文字过于安全，像谁都可以说的正确话
- 开头抓不住人，结尾没有落点
